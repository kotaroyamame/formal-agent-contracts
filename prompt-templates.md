# Formal Agent Contracts — プロンプトテンプレート集

ユーザーが最初にClaudeに渡すプロンプトの雛形。
`{...}` の部分を自分のドメインに書き換えて使う。

---

## テンプレート 1: シンプル（単一エージェント）

最小限の情報で始めたい場合。Claudeが対話的に深掘りしてくれる。

```
{エージェント名}エージェントを定義して。

【扱うデータ】
- {エンティティ名}には{フィールド1}・{フィールド2}・{フィールド3}がある
- {ステータスや列挙型があれば: ステータスは{値1}/{値2}/{値3}}

【やりたい操作】
- {操作1の説明}
- {操作2の説明}

【守りたいルール】
- {ビジネスルール1}
- {ビジネスルール2}
```

### 記入例

```
在庫管理エージェントを定義して。

【扱うデータ】
- 商品には商品ID・商品名・在庫数・カテゴリがある
- カテゴリは食品/日用品/家電

【やりたい操作】
- 入荷（指定数を在庫に加算）
- 出荷（指定数を在庫から減算）
- 在庫照会（商品IDで現在の在庫数を返す）

【守りたいルール】
- 在庫数は0未満にならない
- 出荷数は現在の在庫数を超えられない
- 商品名は空文字を許可しない
```

## テンプレート 2: マルチエージェント（境界契約重視）

複数エージェントが協調するシステム。エージェント間の依存関係を明示する。

```
以下のマルチエージェントシステムの契約を定義して。

【システム概要】
{システムが何をするか1〜2文}

【エージェント構成】
1. {Agent A名} — {役割の説明}
2. {Agent B名} — {役割の説明}
3. {Agent C名} — {役割の説明}

【エージェント間の依存】
- {Agent A}の{操作X}の完了後に{Agent B}の{操作Y}が呼ばれる
- {Agent B}は{Agent C}の{操作Z}を呼び出して結果を受け取る

【共有するデータ型】
- {型名}: {フィールドの説明}

【各エージェントの主要ルール】
- {Agent A}: {制約}
- {Agent B}: {制約}
- {Agent C}: {制約}
```

### 記入例

```
以下のマルチエージェントシステムの契約を定義して。

【システム概要】
ECサイトの注文処理。注文→在庫引当→決済の3エージェントが協調する。

【エージェント構成】
1. OrderAgent — 注文の受付・管理。顧客からの注文を受けて在庫と決済を呼び出す
2. InventoryAgent — 在庫の引当と解放。注文確定時に在庫を減らし、キャンセル時に戻す
3. PaymentAgent — 決済処理。与信確認と売上確定を行う

【エージェント間の依存】
- OrderAgentがConfirmOrderした後、InventoryAgentのReserveStockが呼ばれる
- ReserveStock成功後、PaymentAgentのChargeが呼ばれる
- Charge失敗時はInventoryAgentのReleaseStockで在庫を戻す

【共有するデータ型】
- OrderId: 注文の一意識別子
- ProductItem: 商品ID・数量のペア
- OrderStatus: Pending / Confirmed / Paid / Cancelled

【各エージェントの主要ルール】
- OrderAgent: Paid状態の注文はキャンセルできない
- InventoryAgent: 在庫数は0未満にならない。引当数は現在庫を超えない
- PaymentAgent: 与信確認済みでないと売上確定できない
```

## テンプレート 3: 一気通貫（統合ワークフロー）

定義から検証・証明・コード生成まで一発で回したい場合。

```
統合ワークフローで{システム名}を開発して。

【ドメイン】
{何のためのシステムか}

【データ】
- {エンティティとそのフィールド}

【操作】
- {操作名}: {何をするか}（{事前条件があれば}）

【絶対に守るルール】
- {不変条件・ビジネスルール}

【生成言語】
{TypeScript / Python}
```

### 記入例

```
統合ワークフローで予約管理システムを開発して。

【ドメイン】
会議室の予約管理。ダブルブッキングを仕様レベルで防ぎたい。

【データ】
- 会議室: 室ID・名前・定員
- 予約: 予約ID・室ID・開始時刻・終了時刻・予約者名
- 時刻は自然数（UNIXタイムスタンプ想定）

【操作】
- CreateReservation: 新規予約を作成（同じ部屋の時間帯が重複していないこと）
- CancelReservation: 予約をキャンセル（予約が存在すること）
- ListByRoom: 指定部屋の予約一覧を返す

【絶対に守るルール】
- 同一部屋で時間帯が重なる予約は存在できない（不変条件）
- 開始時刻 < 終了時刻
- 定員は1以上

【生成言語】
TypeScript
```

## テンプレート 4: 既存仕様の形式化

すでに自然言語の仕様書やAPI定義がある場合。

```
以下の仕様をVDM-SLの形式仕様に変換して。

---
{既存の仕様書、API定義、あるいはインターフェース定義をそのまま貼り付け}
---

特に以下の点を形式化してほしい:
- {曖昧になりがちな箇所1}
- {曖昧になりがちな箇所2}

形式化したら検証まで実行して。
```

---

## プロンプトを書くときのコツ

1. **「守りたいルール」を具体的に書く** — これが事前条件・事後条件・不変条件の元になる。「在庫は0未満にならない」のように、境界を数値で示すとClaudeが精度の高い仕様を生成できる

2. **エージェント間の呼び出し順序を書く** — マルチエージェントでは「A.postの後にB.preが呼ばれる」が契約の核心。この依存関係を自然言語で明示すると、A.post ⇒ B.pre の検証が可能になる

3. **最初から完璧を目指さなくていい** — テンプレート1のミニマル版で始めて、Claudeとの対話で深掘りするのが最も効率的。検証フェーズでClaudeが「この境界条件はどうしますか？」と聞いてくる

4. **「統合ワークフローで」と付けると一気通貫** — 定義→検証→証明→コード生成→テストが自動で回る。途中でエラーがあればClaudeが前のフェーズに戻って修正する
