# Proof Obligation (PO) Types — Detailed Reference

Detailed explanations of the major PO types generated by VDMJ, with developer-friendly descriptions.

VDMJが生成する主要なPO種別を、開発者向けの平易な説明とともに解説する。

## 1. subtype obligation (部分型義務)

**Trigger**: When a value must satisfy a type's invariant constraint.

**発生条件**: 値が型の不変条件を満たすか確認が必要な場合

**Developer explanation**: "When assigning or returning this value, does it actually satisfy the type's constraints?"

**開発者向け説明**: 「この値を代入/返却するとき、型の制約をちゃんと満たしている？」

**Example**:
```vdm-sl
types
  Age = nat inv a == a <= 150;

functions
  newborn: () -> Age
  newborn() == 0;  -- PO: Does 0 satisfy Age (0 ≤ x ≤ 150)?
```

Generated PO:
```
inv_Age(0)
-- i.e.: 0 <= 150  → trivially true
```

## 2. invariant satisfiability (不変条件充足可能性)

**Trigger**: When a type or state has an invariant defined.

**発生条件**: 型や状態に不変条件が定義された場合

**Developer explanation**: "Is this constraint contradictory? Does at least one value exist that satisfies it?"

**開発者向け説明**: 「この制約は矛盾していない？少なくとも1つ条件を満たす値が存在する？」

**Example**:
```vdm-sl
types
  Score = nat inv s == s >= 0 and s <= 100;
```

Generated PO:
```
exists s : Score & s >= 0 and s <= 100
```

If written as `inv s == s > 100 and s < 0`, this PO would be unprovable (contradictory constraint).

もし `inv s == s > 100 and s < 0` と書いたら、このPOは証明不能になる（矛盾した制約）。

## 3. map apply obligation (写像適用義務)

**Trigger**: When a map is applied with a key, e.g., `m(k)`.

**発生条件**: `m(k)` のように写像をキーで引く場合

**Developer explanation**: "You're looking up this key in a map — is the key guaranteed to exist?"

**開発者向け説明**: 「このキーでmapを引こうとしてるけど、キーが存在することは保証されてる？」

**Example**:
```vdm-sl
findUser(uid, users) == users(uid)
pre uid in set dom users;
```

Generated PO:
```
forall uid:UserId, users:map UserId to User &
  pre_findUser(uid, users) => uid in set dom users
```

With the pre-condition present, this is trivially satisfied.

事前条件があるため、この場合は自明に成立する。

## 4. total function obligation (全域関数義務)

**Trigger**: Whether pre-conditions, post-conditions, and invariant expressions themselves evaluate correctly.

**発生条件**: 事前条件、事後条件、不変条件の式自体が正しく評価できるか

**Developer explanation**: "Can the condition expression itself cause an error? (e.g., division by zero, out-of-range access)"

**開発者向け説明**: 「条件式そのものがエラーを起こさない？（例: 0除算、範囲外アクセス）」

**Example**:
```vdm-sl
inv u == u.age <= 150
```

Generated PO:
```
forall u:User! & is_(inv_User(u), bool)
```

Confirms the invariant evaluation result is of type bool (no errors during evaluation).

不変条件の評価結果がbool型になること（評価中にエラーが起きないこと）を確認。

## 5. func post condition obligation (関数事後条件義務)

**Trigger**: When a function has a post-condition defined.

**発生条件**: 関数に事後条件が定義されている場合

**Developer explanation**: "Does this function's implementation actually satisfy the declared post-condition?"

**開発者向け説明**: 「この関数の実装は、宣言した事後条件を本当に満たす？」

**Example**:
```vdm-sl
abs: int -> nat
abs(x) == if x >= 0 then x else -x
post RESULT >= 0;
```

Generated PO:
```
forall x:int &
  (let RESULT = if x >= 0 then x else -x in
    RESULT >= 0)
```

## 6. operation postcondition obligation (操作事後条件義務)

**Trigger**: When an operation has a post-condition defined.

**発生条件**: 操作に事後条件が定義されている場合

**Developer explanation**: "After executing this operation, is the promised condition satisfied?"

**開発者向け説明**: 「この操作を実行した後、約束した条件は満たされる？」

Operation POs express state transitions through nested let expressions:

操作のPOはlet式のネストで状態遷移を表現する:
```
forall name, email, mk_State(users, nextId) &
  pre_Op(...) =>
  (let uid = nextId in
    (let users = users munion {uid |-> ...} in
      (let nextId = nextId + 1 in
        (let RESULT = uid in
          postcondition))))
```

## 7. state invariant obligation (状態不変条件義務)

**Trigger**: When an operation modifies state and the state has an invariant.

**発生条件**: 状態を変更する操作があり、状態に不変条件がある場合

**Developer explanation**: "At each step of the operation, is the state's consistency preserved?"

**開発者向け説明**: 「操作の各ステップで、状態の整合性が壊れていない？」

Generated per assignment statement within the operation. Three assignments produce up to three state invariant POs.

操作内の各代入文ごとに生成される。3つの代入があれば、最大3つの状態不変条件POが生成される。

## 8. map compatible obligation (写像互換性義務)

**Trigger**: When using `munion` (map merge).

**発生条件**: `munion`（写像結合）を使用する場合

**Developer explanation**: "When merging two maps, are there any keys mapped to different values?"

**開発者向け説明**: 「2つのmapを合体させるとき、同じキーに異なる値が割り当てられていない？」

```
forall ldom1 in set dom m1, rdom2 in set dom m2 &
  ldom1 = rdom2 => m1(ldom1) = m2(rdom2)
```

## 9. state init obligation (状態初期化義務)

**Trigger**: When a state has an `init` clause.

**発生条件**: 状態にinit句がある場合

**Developer explanation**: "Does the initial value satisfy the state's invariant?"

**開発者向け説明**: 「初期値は状態の不変条件を満たしている？」

## 10. sequence apply obligation (シーケンス適用義務)

**Trigger**: When a sequence is indexed, e.g., `s(i)`.

**発生条件**: `s(i)` のようにシーケンスをインデックスで引く場合

**Developer explanation**: "Is this index within the sequence's bounds?"

**開発者向け説明**: 「このインデックスはシーケンスの範囲内？」

## 11. non-empty sequence obligation (非空シーケンス義務)

**Trigger**: When using `hd s` or `tl s`.

**発生条件**: `hd s` や `tl s` を使用する場合

**Developer explanation**: "You're not calling head/tail on an empty list, right?"

**開発者向け説明**: 「空のリストに対してhead/tailを呼んでいない？」

## 12. recursive function obligation (再帰関数義務)

**Trigger**: When a recursive function is defined.

**発生条件**: 再帰関数がある場合

**Developer explanation**: "Does this recursion always terminate? (No infinite loops?)"

**開発者向け説明**: 「この再帰は必ず終了する？（無限ループにならない？）」
