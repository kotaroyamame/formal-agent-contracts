複数の Claude Code エージェントを並列で走らせている方、こういう経験はないでしょうか。

- エージェントAが作った API を、エージェントBが微妙に違うシグネチャで呼んでいる
- CLAUDE.md に書いたはずの規約が、タスクの受け渡しのたびに少しずつ薄れていく
- 生成速度にレビューが追いつかず、気づけば人間の目が最後の砦になっている

私はあります。マルチコンポーネントのシステムを複数エージェントで並行開発していて、結合したらメッセージ型の名前が食い違っていた、という統合バグを何度か踏みました。エージェントはそれぞれ真面目に仕事をしていたのに、です。

この現象を "constraint drift" と呼ぶ論文も出ています[^1]。制約は主張しただけでは維持されない。エージェント間の受け渡しで劣化していく。プロンプトや CLAUDE.md に書いた約束は、**機械的には何も担保されていない**んですよね。

:::message
この記事は、筆者が開発した Claude Code プラグイン [formal-agent-contracts](https://github.com/kotaroyamame/formal-agent-contracts) の紹介記事です。作者による宣伝が含まれる前提で読んでください。実行ログの数値はすべて実測です（長い出力は抜粋しています）。
:::

[^1]: ["Safe Multi-Agent Behavior Must Be Maintained, Not Merely Asserted"](https://arxiv.org/abs/2605.10481) (arXiv:2605.10481, 2026)

## 仕様駆動開発は「半分」解決した

2025年から2026年にかけて、仕様駆動開発（SDD）が一気に来ました。Kiro、GitHub Spec Kit、cc-sdd。「コードを書く前に仕様を書き、エージェントに仕様を守らせる」という文化は、もう定着しつつあると思います。これは本当に良い変化です。vibe coding の反動として、健全な方向に振れた。

ただ、使い込むほど気になってくることがあります。spec.md は散文なんです。

| | 人間が読める | エージェントが読める | 機械が検証できる |
|---|:---:|:---:|:---:|
| spec.md（散文） | ✅ | ✅ | ❌ |
| 型定義 | ✅ | ✅ | 型までは ✅ |
| テスト | ✅ | ✅ | 書いたケースは ✅ |

ここで「zod とテストで足りるのでは？」と思った方へ。zod が守れるのは*データの形*です。「Done になったタスクは Todo に戻せない」のような、操作をまたぐ状態遷移の整合性はスキーマでは書けません。fast-check のような property-based testing なら性質を書けますが、性質はコードのあちこちに散らばり、性質同士が矛盾していても誰も検出してくれません。この記事で紹介するのは、その2つを1つのソースにまとめて、**矛盾そのものをチェッカーに検査させる**レイヤーです。

散文の仕様は、人間も読めるしエージェントも読めます。でも「実装がこの仕様を満たしているか」を機械はチェックできません。判定しているのは結局、LLM の自己申告（「仕様に沿って実装しました」）と、人間のレビューです。

エージェントが1体ならそれでも回ります。でも並列になると、冒頭のあるあるが起きる。仕様は共有されているのに、解釈が受け渡しのたびにズレていく。ズレを検出する機械がどこにもいない。

つまり SDD は「仕様を書く」問題を解決しましたが、「仕様を守らせる」問題は半分残っています。この記事はその残り半分の話です。

## 残り半分：契約を、機械が判定できる形で書く

やることはシンプルです。エージェント間のインターフェースにあたる部分だけ、散文ではなく、機械が判定できる形式で書きます。

先に言っておくと、**この形式仕様を書くのは Claude です**。人間は日本語で要件を言うだけ。

形式の正体は VDM-SL という ISO 標準の形式仕様記述言語で、型不変条件・事前条件・事後条件を機械が判定できる形で書けます。チェッカー（VDMJ）が構文・型・証明責務を自動検査します。

……と書くと、途端に敷居が上がって聞こえます。形式手法はずっと「難しくて書けない」で知られてきた技術で、VDM-SL を自分で書ける開発者は、普通のチームにはまずいません。だからこそ LLM が記法の習得を肩代わりする今の状況が面白いんです。しかも逆方向も成り立ちます。LLM が書いた仕様を機械が検証するので、LLM の「たぶん大丈夫です」を信じる必要がなくなる。この組み合わせが本命だと私は思っています。

コミュニティマーケットプレースに載っているので、2行で入ります。

```sh
/plugin marketplace add anthropics/claude-plugins-community
/plugin install formal-agent-contracts@claude-community
```

### ステップ1：日本語 → 形式契約

Claude Code でこう打ちます。

```text
タスク管理エージェントを定義して。
- タスクには ID・タイトル・状態（Todo/InProgress/Done）がある
- Done になったタスクは Todo/InProgress に戻せない
- タイトルは100文字以内
```

define-contract スキルが対話で要件を深掘りします。たとえば「Done のタスクのタイトルは編集できますか？」のような、書き忘れた境界を聞き返してくる。答えていくと、こういう VDM-SL が出てきます（記事用に簡略化しています）。

```text:task.vdmsl（抜粋・簡略化）
Task :: id : TaskId  title : seq1 of char  status : Status
inv t == len t.title <= 100;

ValidTransition: Status * Status -> bool
ValidTransition(fromSt, toSt) ==
  if fromSt = toSt then true
  else if fromSt = <Done> then false   -- Done からは戻れない
  else true;
```

「タイトルは100文字以内」が不変条件に、「Done から戻せない」が遷移関数に、そのまま形式になっているのが見えると思います。生成した仕様が VDMJ の型チェックを通らなければ Claude が自分で直すので、人間が VDM-SL をデバッグする場面は基本ありません。レビューしたいときは export-human-spec スキルが日本語の仕様書に逆変換してくれます。

### ステップ2：機械が検証する

「仕様を検証して」と言うと、VDMJ が動きます。同梱サンプル（上のタスク管理の完全版）での実測出力がこれです。

```text:VDMJ の出力（抜粋）
Parsed 1 module in 0.029 secs. No syntax errors
Type checked 1 module in 0.07 secs. No type errors
Generated 37 proof obligations
```

証明責務（この仕様が満たすべき条件）が37個生成されました。大変そうに聞こえますが、全部チェッカーが自動で流します。この仕様では反例ゼロ（FAILED 0。内訳は PROVABLE 16 / MAYBE 19 / TIMEOUT 2 で、MAYBE は「反例が見つからなかった」であって証明ではありません）。人間の出番は、反例が出たときだけです。

で、その反例がどう出るか。試しに、事後条件に off-by-one のバグをわざと仕込んで再検証してみます。

```text:QuickCheck の出力（抜粋）
PO #21, FAILED in 0.001s
Counterexample:
CreateTask: operation postcondition obligation ...
 --> board = {|->}, nextId = 1, title = "a"
 ...
 --> returns false
```

nextId が 1 のとき戻り値は 1 になるはずが、仕込んだ事後条件は 0 を要求している。この矛盾が、コードを1行も書く前に、具体的な反例つきで返ってきました。

ここがこの仕組みの核心です。**仕様のバグや矛盾が、実装前に、LLM の感想ではなくチェッカーの判定として検出される。** 並列エージェントの世界では、この「判定が機械であること」の価値がどんどん上がっていきます。

### ステップ3：契約からテストとコードが出てくる

検証済みの契約からは、Vitest/Jest の契約テストが自動生成されます（generate-tests スキル。以下は出力イメージ）。

```typescript
it("rejects Done -> Todo transition", () => {
  expect(() => mgr.changeStatus(doneTaskId, "Todo"))
    .toThrow(ContractError);   // 事前条件違反
});
```

境界値テスト（100文字ちょうど/101文字）も同時に出ます。生成される TypeScript/Python のスキャフォールドにはランタイム契約チェックが埋め込まれているので、別のエージェントが契約を破るコードを書けば、契約テストとランタイムチェックに触れた時点で落ちます。生成されたケースの範囲で、ではありますが、プロンプトの言いつけよりはるかに硬い。冒頭の constraint drift を、注意力ではなくテストランナーが止める構図です。

ちなみに、冒頭で書いた「メッセージ型名の食い違い」は、この構図では起きようがありません。エージェントAもBも同じ .vdmsl から型を導出するので、食い違う自由がそもそもないからです。

なお、ここまで（契約定義とテスト生成）は外部ツール不要です。VDMJ と Z3 が要るのはステップ2のフル検証だけで、それも Java 11 以上の環境なら、配布 zip の展開と `pip install z3-solver` で済みます。生成したテストを回すのに使うのは、みなさんのプロジェクトに既にある Node.js + Vitest/Jest です。

## これは「証明できるガードレール」

t_wada さんが対談で「AIエージェント時代に、TDD はガードレールになる」という話をされています[^2]。この整理が私はとても好きで、形式契約はその延長線上に置けます。

[^2]: [AIエージェント時代に、テスト駆動開発（TDD）は「ガードレール」になる](https://agilejourney.uzabase.com/entry/2025/08/29/103000)（t_wada × やっとむ対談, Agile Journey, 2025）

| レイヤー | 担保するもの | 誰が判定するか |
|---|---|---|
| spec.md | 意図の共有 | 人間 + LLM |
| テスト（TDD） | 書いたケースの正しさ | テストランナー |
| 形式契約 | 宣言した性質の整合性 | VDMJ + Z3 |

型は形しか守れません。テストは書いたケースしか守れません。形式契約は「すべての入力に対して成り立つべき性質」を宣言し、チェッカーがその矛盾を探します。Z3 で証明まで通ればその性質は全入力で保証され、QuickCheck 段階でも高強度のサンプリング検査になります。

そして重要なのは、契約という**1つのソース**から、型・テスト・ランタイムチェックが全部生成されることです。要件が変わったら直すのは .vdmsl 1ファイルで、下流はそこから再生成します（生成物には手を入れない運用です。コード側を先に変えてしまったときは reconcile-code スキルが契約との差分を出します）。契約とコードが乖離したまま放置されると何が起きるかというと、契約テストが落ち続けるだけです。spec.md が静かに腐るのに対して、形式契約は腐り方がうるさい。実務ではこの差がいちばん効くと思っています。

## なぜ VDM なのか

形式手法には TLA+ も Alloy も Dafny もあります。VDM を選んだ理由は2つです。

ひとつは用途です。エージェント間の「データと操作の契約」を書くには、豊富なデータ型と事前・事後条件を持つ VDM-SL が素直でした。TLA+ は並行システムの時相的な設計に、Dafny は検証済み単体コードに寄っています。

もうひとつは日本との縁です。おサイフケータイやモバイル Suica に載っているモバイル FeliCa IC チップのファームウェア仕様は、VDM++ で記述されていました。スマホで改札を通るたび、形式手法で書かれた仕様が動いているんですよね。日本語の教育資産や文献も、形式手法の中では豊富な部類です。

ちなみに、私の調べた限り、VDM と AI コーディングエージェントを繋いだ公開ツールはまだこれひとつです（TLA+/Lean/Dafny にはエージェント統合が既にありますが、VDM は空白でした）。空白ということは、需要もまだ証明されていないということです。「誰もいない場所に立っている」という現状報告として書いておきます。

## 仕様の先へ：DB スキーマも契約から導出する

直近の更新で、仕様が確定した後の一番重いフェーズ（設計）に一歩踏み込みました。第一弾は DB スキーマの導出です。VDM-SL の型と不変条件から、こういう DDL が出てきます（同梱サンプルの実出力から抜粋）。

```sql:schema.sql（抜粋）
title  VARCHAR(100) NOT NULL,  -- seq1 of char, inv: len title <= 100 (R5, R7)
status task_status   NOT NULL, -- Status = <Todo> | <InProgress> | <Done> (R3)
-- 「Done から戻れない」は CHECK では書けない → UPDATE トリガーで担保 (R12)
```

ポイントは、LLM に「それらしいスキーマ」を自由作文させず、番号付きの写像規則で導出することです。そして、ここが形式手法らしいところなんですが、**規則で写せないものを隠しません**。`nat`（上限なし）は `BIGINT`（有限）にしか写せない。複数行にまたがる不変条件は CHECK 制約で書けない。こうした「仕様と実装の乖離」を DEVIATIONS.md という表に全部記録して、それぞれに具体的な補償手段（トリガー / アプリ層の契約チェック / 契約テスト）を割り当てます。あわせて「各不変条件はどこで守られるのか」の対照表（TRACEABILITY.md）を作り、担保箇所のない不変条件が残っていたら導出は未完成として扱います。

乖離を書き出すという行為は地味ですが、本質だと思っています。どんな実装も仕様を完全には満たせません。問題は乖離があることではなく、乖離が**どこにあるか誰も知らない**ことです。ちなみに冒頭の型名食い違いバグは、このプラグインの初期版を自分で使っていた頃に踏んだもので、それが設計文書サポート（v2.0）を作る直接のきっかけになりました。導出の詳細は長くなるので、次の記事で書きます。

## 正直な限界の話

作者として、できないことも書いておきます。

**チェッカーが保証するのは仕様の内部整合性で、あなたの意図との一致ではありません。** Claude が「間違っているが矛盾はしていない」仕様を書けば、VDMJ は通します。ここを守るのは、export-human-spec で逆変換した日本語仕様書への人間のレビューです。この最後の照合だけは、機械に渡せません。

**VDM-SL で表現できない仕様があります。** UI の使い心地、性能、セキュリティ要件。形式契約が担当するのはドメインロジックとデータ契約です。UI フローの E2E テスト導出はロードマップに入れていますが、現時点では守備範囲外です。

**効果の評価はまだ探索的です。** プラグインあり/なしで同一課題を比較した評価では仕様カバレッジ +42.8pt という差が出ていますが、鵜呑みにしないでください。

:::message
この数値は、同一モデルが両群を生成しキーワードベースで採点した探索的評価（3課題 × 5試行 × 2条件）によるもので、有意性検定は行っていません。詳細と限界は[リポジトリの評価レポート](https://github.com/kotaroyamame/formal-agent-contracts/tree/master/eval)を参照してください。
:::

**VDM-SL を読める人はチームに増えません。** 読む必要が薄くなるように作ってはいますが（生成・修正・解説は Claude、レビューは日本語仕様書）、最後の最後で形式仕様を精読すべき場面が来たら、それは学習コストです。

## おわりに：何を機械に渡すか

この記事自体、機械が検証できたのは実行ログの部分だけです。残りの検証はあなたに委ねられている。spec.md と同じ限界です。

だからこそ、と私は思うんです。検証できる部分だけでも、機械に渡す価値がある。全部は渡せません。意図の共有には散文が要るし、判断には人間が要る。でも「Done から Todo に戻せない」のような、機械が判定できる約束まで人間の注意力で守るのは、もうやめてもいい。

エージェントの数がこれからも増えていくとして、何を機械の判定に渡し、何を人間の判断に残すのか。この線引きが、AI 駆動開発の次の設計課題だと思っています。あなたのチームでは、どこに線を引きますか。
