# Formal Agent Contracts — 評価プロトコル

## 1. 評価の目的

Formal Agent Contracts プラグインを使用した場合と使用しなかった場合で、
AI駆動エージェント開発の成果物の品質にどの程度の差が生じるかを定量的に計測する。

### 評価仮説

- **H1**: プラグイン使用群は、非使用群に比べて契約違反（事前条件・事後条件・不変条件の破れ）の検出率が有意に高い
- **H2**: プラグイン使用群は、非使用群に比べて境界条件の仕様カバレッジが有意に高い
- **H3**: プラグイン使用群は、非使用群に比べて仕様とコードの乖離（仕様漂流）が有意に少ない

---

## 2. 実験設計

### 2.1 実験条件

| 条件 | 説明 |
|------|------|
| **A群（Control）** | Claude Code のみ。形式手法ツールなし。自然言語での指示のみ |
| **B群（Treatment）** | Claude Code + Formal Agent Contracts プラグイン |

### 2.2 ベンチマーク課題

同一の開発課題を A群・B群 それぞれで実施する。
課題は3つ、複雑度を段階的に上げる:

#### Task 1: 単一エージェント（Low Complexity）

**課題: 銀行口座管理エージェント**

```
銀行口座エージェントを開発してください。

機能:
- 口座開設（名義、初期残高）
- 入金（金額を残高に加算）
- 出金（金額を残高から減算）
- 残高照会
- 口座間送金（送金元から減算、送金先に加算）

ビジネスルール:
- 残高は0未満にならない
- 1回の出金上限は100万円
- 口座名義は空文字不可
- 送金は送金元と送金先が異なること
```

**埋め込みトラップ（評価用）:**
- 送金時のアトミック性: 送金元の減算後・送金先の加算前に失敗した場合の処理
- 初期残高が負の場合の扱い（明示されていない）
- 同一口座への送金（エッジケース）
- 出金上限と送金の関係（送金にも上限が適用されるか？）
- 残高0円の口座からの0円出金（境界値）

#### Task 2: マルチエージェント（Medium Complexity）

**課題: 図書館貸出管理システム**

```
図書館の貸出管理システムを3つのエージェントで開発してください。

エージェント構成:
1. CatalogAgent — 書籍の登録・検索・在庫管理
2. MemberAgent — 会員の登録・貸出資格管理
3. LoanAgent — 貸出・返却・延滞管理

ビジネスルール:
- 会員は最大5冊まで同時貸出可能
- 延滞中の会員は新規貸出不可
- 在庫が0の書籍は貸出不可
- 貸出期間は14日。延長は1回まで（追加7日）
- 書籍のカテゴリは「一般」「参考」「貴重」の3種
- 貴重書籍は館内閲覧のみ（貸出不可）
```

**埋め込みトラップ（評価用）:**
- 貸出中の書籍を別の会員が予約できるか？（明示されていない）
- 延滞中に返却した直後、同じ会員が即座に別の本を借りられるか？
- 5冊貸出中に1冊返却→即座に別の本を借りる際のカウント
- CatalogAgent と LoanAgent の在庫数の整合性
- 延長済みの本の返却遅延の扱い
- 同一書籍の複数コピーの管理

#### Task 3: 状態遷移 + 並行性（High Complexity）

**課題: オンラインオークションシステム**

```
オンラインオークションシステムを開発してください。

機能:
- 出品（商品名、開始価格、期間、出品者ID）
- 入札（オークションID、入札額、入札者ID）
- 落札（期間終了時に最高額入札者が落札）
- 支払い処理
- 商品発送確認

状態遷移: Draft → Active → Ended → Paid → Shipped → Completed
         Active → Cancelled（入札がない場合のみ）

ビジネスルール:
- 入札額は現在の最高額より高くなければならない
- 出品者自身は入札できない
- オークション期間中のみ入札可能
- 終了5分前に入札があった場合、終了時刻を5分延長
- 最低入札単位は現在価格の1%（最低100円）
- 落札後72時間以内に支払いがない場合は自動キャンセル
```

**埋め込みトラップ（評価用）:**
- 終了5分前の入札による延長が連鎖した場合の上限
- 同時刻の入札が複数あった場合の優先順位
- 落札キャンセル後の再出品 or 次点落札の扱い
- Cancelled状態からの再出品可否
- 入札者が入札を取り消せるか（明示されていない）
- 最低入札単位の端数処理

### 2.3 実施手順

```
1. A群: 各課題を Claude Code のみで実施
   プロンプト: 「{課題文}。TypeScriptで実装してください。テストも書いてください。」

2. B群: 各課題を Claude Code + プラグインで実施
   プロンプト: 「統合ワークフローで{課題文}を開発して。TypeScriptで生成して。」

3. 各条件の出力を収集:
   - 生成されたソースコード
   - 生成されたテスト
   - （B群のみ）VDM-SL仕様、POリスト、検証結果
```

### 2.4 制御変数

実験の公平性を確保するために以下を統一する:

- Claude のモデルバージョン（同一バージョンを使用）
- プロンプトの情報量（課題文は両群同一。B群に追加情報を与えない）
- 実行環境（同一マシン、同一ランタイム）
- 試行回数: 各課題×各条件で **5回** 実施（LLMの出力の分散を吸収）

## 3. 評価メトリクス

### 3.1 M1: 契約違反検出率（Contract Violation Detection Rate）

**定義**: 事前に定義した「埋め込みトラップ」（エッジケース）のうち、
生成されたコードまたはテストが検出・対処しているものの割合。

```
CVDR = (検出されたトラップ数) / (全トラップ数) × 100%
```

**計測方法**:
各トラップに対応するテストケースを評価者が事前に作成（ゴールドスタンダード）。
生成されたコードにそのテストケースを実行し、以下を判定:

| 結果 | スコア |
|------|--------|
| コードが正しく処理し、テストも存在する | 3 (Full) |
| コードが正しく処理するが、テストがない | 2 (Partial) |
| コードが正しく処理しないが、テスト(or仕様)で言及がある | 1 (Aware) |
| 完全に未対処 | 0 (Missing) |

### 3.2 M2: 仕様カバレッジ（Specification Coverage）

**定義**: 課題のビジネスルールのうち、
コード内で明示的に検証（バリデーション、アサーション、型制約）されているものの割合。

```
SC = (明示的に検証されているルール数) / (全ビジネスルール数) × 100%
```

**計測方法**:
各ビジネスルールに対して、以下のいずれかが存在するかを確認:

| 検証形態 | スコア |
|----------|--------|
| ランタイム契約チェック（pre/post/inv） | 3 |
| if-throw / assert 等の明示的バリデーション | 2 |
| 型制約のみ（enum, branded type等） | 1 |
| 検証なし | 0 |

### 3.3 M3: 境界条件網羅度（Boundary Condition Coverage）

**定義**: テストコードが境界値をどの程度網羅しているか。

各ビジネスルールについて、以下の境界値テストの有無を確認:

- **ちょうど境界値** (e.g., 残高=0で出金0円)
- **境界値-1** (e.g., 残高=99で出金100円)
- **境界値+1** (e.g., 残高=101で出金100円)
- **型の境界** (e.g., 空文字列、null、負数)

```
BCC = (存在する境界値テスト数) / (全境界値テスト候補数) × 100%
```

### 3.4 M4: 仕様明示性（Specification Explicitness）

**定義**: ビジネスルールが自然言語ではなく、
機械的に検証可能な形式で記述されている度合い。

| レベル | 定義 | スコア |
|--------|------|--------|
| L3: 形式仕様 | VDM-SL等の形式言語で定義 | 3 |
| L2: コード内契約 | TypeScript型、assert、バリデーション | 2 |
| L1: コメント/ドキュメント | JSDoc、README等で言及 | 1 |
| L0: 暗黙 | どこにも記述がない | 0 |

```
SE = Σ(各ルールのスコア) / (ルール数 × 3) × 100%
```

### 3.5 M5: コード-仕様一致率（Code-Spec Alignment）

**定義**: 仕様（自然言語、コメント、または形式仕様）に記述された内容と、
実際のコードの振る舞いが一致している割合。

**計測方法**:
生成されたコードの各操作について、仕様との整合性を評価者が判定する:

| 状態 | スコア |
|------|--------|
| 仕様とコードが完全に一致 | 2 |
| 仕様はあるが、コードが部分的に異なる | 1 |
| 仕様とコードが矛盾 / 仕様がない | 0 |

### 3.6 M6: テスト有効性（Test Effectiveness）

**定義**: 生成されたテストが、意図的に埋め込んだバグを検出できる割合。

**計測方法**:
ゴールドスタンダードのコードに対して、以下のミューテーションを適用:
- 境界条件の変更 (`<` → `<=`)
- ガード条件の削除
- 状態遷移の条件除去
- 戻り値の改変

```
TE = (テストが検出したミューテーション数) / (全ミューテーション数) × 100%
```

## 4. ゴールドスタンダード（正解基準）

各課題について、評価者が事前に以下を定義する:

### 4.1 完全仕様書

課題文に明示されていない暗黙の仕様も含めた完全な仕様。
「埋め込みトラップ」に対する正解の振る舞いを定義。

### 4.2 ゴールドテストスイート

全トラップに対応するテストケース。
生成されたコードの品質を客観的に測定するために使用。

### 4.3 ミューテーション定義

M6（テスト有効性）の計測に使用するコード変異のリスト。

## 6. 統計分析

### 6.1 サンプルサイズ

- 3課題 × 5試行 × 2条件 = 30データポイント
- 課題ごとの分析: 5試行 × 2条件 = 10データポイント/課題

### 6.2 検定方法

- **条件間比較**: Wilcoxon順位和検定（ノンパラメトリック、サンプルサイズ小）
- **効果量**: Cliff's delta（ノンパラメトリック効果量）
  - |d| < 0.147: 効果なし
  - |d| < 0.33: 小
  - |d| < 0.474: 中
  - |d| ≥ 0.474: 大
- **信頼区間**: ブートストラップ法 (B=1000)

### 6.3 レポート形式

```markdown
## 結果サマリー

| メトリクス | A群 (Control) | B群 (Treatment) | Δ | p値 | 効果量 |
|-----------|--------------|-----------------|-----|-----|--------|
| M1: 契約違反検出率 | 35.2% ± 8.1 | 78.6% ± 5.3 | +43.4pp | <0.01 | 0.92 (大) |
| M2: 仕様カバレッジ | 62.0% ± 11.2 | 91.4% ± 4.8 | +29.4pp | <0.01 | 0.88 (大) |
| ...
```

## 7. 脅威と対策（Threats to Validity）

### 内的妥当性

| 脅威 | 対策 |
|------|------|
| LLMの出力のランダム性 | 各条件5回試行で分散を吸収 |
| プロンプトの差による不公平 | 課題文は両群完全同一。B群に追加情報を与えない |
| 評価者バイアス | M1/M5 は二重盲検（条件ラベルを伏せて評価） |
| ゴールドスタンダードの恣意性 | 課題設計前にトラップリストを固定。事後追加しない |

### 外的妥当性

| 脅威 | 対策 |
|------|------|
| 課題の代表性 | 3段階の複雑度で幅を確保 |
| 実務との乖離 | 課題は実プロジェクトで頻出するパターンに基づく |
| 特定モデルへの依存 | モデルバージョンを明記。他モデルでの追試を推奨 |

### 構成概念妥当性

| 脅威 | 対策 |
|------|------|
| メトリクスが「品質」を正しく捉えているか | 6つの独立したメトリクスで多面的に評価 |
| 形式仕様の存在自体がスコアを押し上げる | M4（仕様明示性）以外は実コードの振る舞いで評価 |
