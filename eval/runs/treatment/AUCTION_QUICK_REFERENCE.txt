EDGE CASE TRAPS (14 total)
================================================================================

 1. Extension Cascade Limit       - Must limit extensions (MAX=10)
 2. Simultaneous Bids             - Total ordering required
 3. Next Bidder on Timeout        - State behavior after cancellation
 4. Relist from Cancelled         - Transition map control
 5. Bid Withdrawal                - Operation must be absent or explicit
 6. Minimum Increment Rounding    - floor() and max() usage
 7. First Bid vs Starting Price   - >= vs > distinction
 8. Exact End Time Boundary       - now < end_time vs now <= end_time
 9. 72-Hour Exact Boundary        - <= vs > for payment window
10. Invalid State Transitions     - Invariant enforcement
11. Draft to Cancelled            - Allowed transition
12. Seller Bidding via Proxy      - bidder_id <> seller_id invariant
13. Zero/Negative Starting Price  - starting_price > 0 validation
14. Zero Duration Auction         - duration > 0 validation

================================================================================
KEY INSIGHTS
================================================================================

1. Formal Specification VALUE:
   VDM-SL forces explicit treatment of boundary conditions that code
   developers often overlook (e.g., now < end_time vs now <= end_time)

2. Invariants as Specification:
   Run5's exhaustive invariants (covering all 14 traps) demonstrate how
   formal systems can encode business logic and constraints

3. Preconditions as Contract:
   Clear preconditions prevent invalid operations at specification time,
   catching errors before code execution

4. Timing is Critical:
   Most traps (8 of 14) involve timing: boundaries, cascades, timeouts.
   Formal specification makes timing explicit and verifiable.

5. No Substitute for Completeness:
   Run4 (10 traps) vs Run5 (14 traps) shows that simplicity can miss
   important edge cases. Comprehensiveness requires explicit specification.
